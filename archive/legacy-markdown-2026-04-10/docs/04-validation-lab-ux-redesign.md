# 전략 검증 랩 UI/UX 개편안

작성일: 2026-04-03

## 한 줄 결론

현재 `운영 콘솔 -> 전략 검증 랩` 은 기능은 많지만,
작업 순서가 화면 구조에 드러나지 않아 사용자가 계속 스크롤을 오가며
설정 / 실행 / 상태 확인 / 다음 액션을 반복해야 한다.

이 화면은 정보 전시형이 아니라 **작업형 콘솔**로 재구성해야 한다.

---

## 1. 현재 문제 요약

### 문제 1. 스크롤 동선이 너무 길다
사용자는 아래를 반복하게 된다.

1. 설정 변경
2. 저장 버튼 클릭
3. 스크롤 내려 상태 확인
4. 다시 내려 결과 확인
5. 다시 올라 다음 버튼 클릭
6. 또 내려 로그/판정 확인

즉 버튼과 상태와 결과가 분리돼 있어 작업 피로가 크다.

### 문제 2. 액션과 상태가 붙어 있지 않다
버튼을 누른 직후 바로 옆에서
- 실행 중인지
- 완료됐는지
- 차단됐는지
- 다음 단계가 뭔지
보여야 하는데 현재는 화면 각 영역에 흩어져 있다.

### 문제 3. 워크플로우가 보이지 않는다
실제 사용 순서는 다음과 같다.

- 설정 저장
- 백테스트
- Search
- Revalidate
- Save
- Runtime Apply

하지만 UI는 이 순서를 한 줄의 작업 흐름으로 보여주지 않는다.

### 문제 4. 로그 비중이 너무 크다
로그 버튼이 별도로 있는데도 메인 화면 하단 로그가 크게 자리 잡고 있다.
메인 본문을 갉아먹는 구조라 작업 집중을 깨뜨린다.

### 문제 5. 가드레일 정책이 백테스트/검증 설정과 분리돼 있다
사용자 체감상 검증 기준과 저장/반영 기준은 한 세트인데,
현재는 멀리 떨어져 있어 사고 흐름이 끊긴다.

### 문제 6. 용어가 헷갈린다
아래 용어가 분리되지 않아 혼란을 만든다.

- 후보 풀
- 실행 후보
- 저장 후보
- runtime 반영 후보
- 학습 / 검증 / 탐색 기간

특히 `학습` 이라는 표현은 현재 구현 의미와 잘 맞지 않는다.

### 문제 7. Revalidate 거절 시 자동 Search 재시작처럼 보이는 UX
정상 의도는 아래여야 한다.

- Search = 후보 풀 탐색
- Revalidate = 이미 나온 search 결과 재판정
- Reject = 저장/반영 차단

즉 Revalidate가 거절됐다고 자동으로 새 Search를 다시 시작하는 것은 바람직하지 않다.
실제 backend가 재탐색하는지, 아니면 UI 상태 표시가 자동 재탐색처럼 보이게 만드는지는 별도 확인이 필요하지만,
사용자 체감상 현재 동작은 잘못된 워크플로우처럼 느껴진다.

### 문제 8. 설정 변경 후 백테스트 결과가 이전 값처럼 보이는 stale 문제
사용자 제보 기준 증상은 아래와 같다.

1. Revalidate 거절 후 백테스트 설정을 변경
2. 저장
3. 다시 백테스트 실행
4. 너무 빨리 완료되거나, 이전 값 기준처럼 보임

이 문제는 정상 UX가 아니다.

의심 원인:
- 저장 후 이전 실행 결과(`executedRun`)가 stale 처리되지 않음
- 백테스트 API가 이전 계산 결과를 즉시 재사용함
- 현재 실행 기준 / 저장 기준 / 마지막 실행 결과가 화면에서 섞여 보임

즉 설정을 바꿨으면 UI는 아래를 반드시 보장해야 한다.
- 이전 결과 stale 표시
- 다시 실행 시 새 계산 진행 중 표시
- 새 계산이 어떤 설정값 기준인지 명시

현재는 이 연결이 끊겨 있어 검증 랩 신뢰도를 떨어뜨린다.

## 2. 개편 목표

이 화면의 목표는 사용자가 아래를 한 번에 이해하게 만드는 것이다.

1. 지금 어떤 기준으로 보고 있는가
2. 다음 버튼은 무엇을 눌러야 하는가
3. 왜 저장/반영이 가능한가 또는 막혔는가
4. 지금 보고 있는 후보가 search 결과인지 실행 후보인지

즉 이 화면은 **읽는 화면**이 아니라 **실행하는 화면**이어야 한다.

## 3. 제안하는 전체 구조

## A. 상단 고정 액션 바
스크롤과 무관하게 항상 보이는 sticky action bar를 둔다.

포함 항목:
- 현재 기준 배지
  - 저장 설정 기준
  - 마지막 백테스트 기준
  - 후보 풀 탐색 기준
- 액션 버튼
  - 저장
  - 백테스트 실행
  - 최적화(Search)
  - 재검증(Revalidate)
  - 저장(Save)
  - Runtime 반영(Apply)
- 현재 상태
  - 대기 / 실행 중 / 완료 / 차단
- 마지막 실행 시각

핵심은 **스크롤해도 액션 바가 사라지지 않게 하는 것**이다.

---

## B. 본문 레이아웃: 좌우 2컬럼

### 좌측 컬럼 = 입력

#### 카드 1. 실행 기준 설정
현재 백테스트/검증 기준을 조정하는 핵심 카드.

포함 항목:
- 시장
- 탐색 기간
- 검증 기간
- 최대 보유일
- RSI / 거래량 / ADX / MFI / BB / Stoch
- stop loss / take profit
- runtime 후보 소스 모드
  - quant_only
  - hybrid

#### 카드 2. 저장/초안 관리
포함 버튼:
- 저장
- 저장값 불러오기
- 기본값 초기화

표시 상태:
- 초안 변경 있음
- 서버 저장값과 동일

#### 카드 3. 가드레일 정책
현재는 분리되어 있지만, 실행 기준 설정과 같은 컬럼에서 보여주는 편이 맞다.

포함 항목:
- 저장 가능 기준
- runtime 반영 가능 기준
- limited adopt 제약
- 차단 기준

즉 사용자 입장에서는 **설정 + 판정 기준**이 한 묶음으로 보여야 한다.

---

### 우측 컬럼 = 현재 상태

#### 카드 4. 현재 실행 기준
이 카드는 저장값/마지막 백테스트 기준만 보여준다.

표시 항목:
- 시장
- 탐색 기간
- 검증 기간
- 전략명
- objective
- runtime 후보 소스 모드
- 기준 시각

#### 카드 5. 현재 후보 풀 탐색 기준
optimizer/search 결과 기준만 보여준다.

표시 항목:
- 시장
- 탐색 기간
- 검증 기간
- top N
- optimizer 버전
- 생성 시각

#### 카드 6. latest candidate 상태
표시 항목:
- decision
- 저장 가능 여부
- runtime 반영 가능 여부
- near miss
- 차단 이유

#### 카드 7. saved/runtime 상태
표시 항목:
- 저장된 후보 ID
- 저장 시각
- runtime 반영 여부
- 반영 시각

핵심은 **실행 기준 카드와 후보 풀 탐색 카드가 섞이지 않게 하는 것**이다.

---

## C. 하단 결과 영역은 탭화
현재처럼 세로로 결과를 길게 늘이지 말고, 탭으로 나눈다.

탭 제안:
1. 백테스트 결과
2. Walk-forward 결과
3. 후보 풀 탐색 결과
4. latest candidate 판정
5. 종목별 후보
6. 로그

이렇게 하면 사용자는 필요한 정보만 열어 볼 수 있고,
본 화면의 세로 길이도 크게 줄어든다.

---

## 4. 버튼 동선 설계

상단 sticky action bar 기준 권장 순서:

1. 저장
2. 백테스트 실행
3. 최적화(Search)
4. 재검증(Revalidate)
5. 저장(Save)
6. Runtime 반영(Apply)

### 버튼 활성화 규칙
- 저장 안 했으면 백테스트/최적화 버튼 비활성 또는 경고
- 백테스트 결과 없으면 Search/Revalidate 비활성
- latest candidate 없으면 Save/Apply 비활성
- guardrail 차단이면 Save/Apply 비활성 + 이유 표시

즉 버튼은 눌러봐야 아는 구조가 아니라,
**왜 지금 못 누르는지 바로 보여주는 구조**가 되어야 한다.

---

## 5. 용어 개편 제안

### 현재 문제가 되는 용어
- 후보
- 실행 후보
- 후보 풀
- latest candidate
- saved candidate
- 학습 기간

### 제안하는 용어
- **후보 풀 탐색 결과** = Search 결과
- **실행 후보** = latest candidate
- **저장된 실행 후보** = saved candidate
- **현재 runtime 기준** = applied candidate
- **탐색 기간** = lookback_days
- **검증 기간** = validationDays

`학습` 은 현재 구현 의미와 맞지 않으므로 UI에서는 가급적 제거하는 편이 낫다.

---

## 6. 로그 UX 개편

현재 로그는 본문에서 너무 큰 비중을 차지한다.

### 권장 방식
- 기본은 최근 3줄만 노출
- `전체 보기` 버튼으로 모달/드로어 열기
- 에러는 상단 배지/인라인 카드로만 요약

즉 로그는 메인 본문을 차지하는 요소가 아니라,
필요할 때만 깊게 보는 보조 수단이 되어야 한다.

---

## 7. Revalidate 거절 동작 원칙

### 기대 동작
Revalidate가 거절되면 UI는 아래 중 하나를 보여줘야 한다.

- `거절`
- 이유
- 다음 액션 선택지
  - 설정 수정
  - 다시 백테스트
  - 다시 Search

즉 사용자가 명시적으로 다시 Search를 누르기 전까지는
자동으로 새 Search를 시작하면 안 된다.

### 현재 문제
사용자 체감상 Revalidate 거절 이후 자동으로 Search가 다시 시작되는 것처럼 보인다.

이 문제는 두 가능성이 있다.
1. backend가 실제로 새 Search를 트리거
2. UI가 search 상태를 다시 활성처럼 보여줘 자동 Search처럼 오해를 유발

어느 쪽이든 UX 상으로는 수정이 필요하다.

### 개선 원칙
- Revalidate 거절은 거절 상태로 고정
- Search 재실행은 사용자 클릭으로만 시작
- 자동 복구/handoff 상태는 별도 작은 배지로만 노출


## 10. 최종 요약

현재 전략 검증 랩의 가장 큰 문제는 기능 부족이 아니라,
**작업 순서가 UI에 보이지 않는다는 점**이다.

즉 개선 방향은
- 기능 추가보다
- 작업 흐름 중심 재배치
- 용어 정리
- 버튼/상태 결합
- 로그 축소
쪽이 맞다.

특히 다음 두 가지는 꼭 반영해야 한다.

1. **실행 기준 카드와 후보 풀 탐색 기준 카드 분리**
2. **Revalidate 거절 시 자동 Search처럼 보이지 않도록 UX 수정**
