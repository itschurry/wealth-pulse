# Codex Handoff — 전략 후보 셋 중심으로 전략 검증 랩 재구성

작성일: 2026-04-03

## 한 줄 요약

`전략 검증 랩`은 원래 **전략 후보 셋을 비교하고 승인/보류/반영하는 화면**이어야 한다.
그런데 현재 구현은 중간에 **종목 후보 개념**이 섞였고, 이후 그 종목 후보 흐름을 제거하는 과정에서,
지금은 **전략 후보 셋 UI/데이터 모델이 반쯤만 맞는 상태**가 되었다.

이 작업의 목표는:
- 종목 후보 개념 없이
- optimizer가 찾은 **전략 후보 셋(strategy candidates)** 을 1급 객체로 저장/노출하고
- 사용자가 그 후보 셋 중 하나를 골라 **재검증 → 저장 → runtime 반영** 할 수 있게 만드는 것이다.

---

## 현재 문제

### 1. 데이터 모델이 전략 후보 셋을 1급 객체로 저장하지 않음
현재 optimizer 저장 구조는 대략 아래와 같다.

- `global_params`
- `per_symbol`
- `meta`

즉 대표 `global_params`는 있지만,
**서로 다른 전략 파라미터 조합 후보들의 리스트**가 정식 구조로 저장되지 않는다.

그래서 프론트에서 `전략 후보 셋`을 보여주려 해도,
지금은 `per_symbol` 등을 억지로 해석해 중간 다리 UI를 만들 수밖에 없는 상태다.

### 2. 현재 붙은 `전략 후보 셋` UI는 완전히 맞는 모델이 아님
지금 프론트에 붙은 후보 리스트는 사용자가 원하는
"독립적인 전략안 후보 목록" 이라기보다,
기존 optimizer 산출물을 끌어다 임시로 표시한 성격이 강하다.

즉 아래 조건을 만족해야 진짜 전략 후보 셋인데:
- 행 하나 = 독립적인 전략안 1개
- 각 후보마다 파라미터 patch / 성능 / 신뢰도 / 점수 / 선택 이유 존재
- 그 후보를 직접 재검증 가능

현재는 이게 완전히 만족되지 않는다.

### 3. 종목 후보 개념은 전략 검증 랩에서 제거하는 방향이 이미 결정됨
사용자 요구사항은 명확하다.

- `전략 검증 랩`에서 종목 후보 흐름 제거
- 종목 승인/보류/저장은 이 화면에서 다루지 않음
- 여기서는 **전략 후보 셋**만 다룸

현재 작업 중 일부 코드에서는 이 방향으로 이미 정리되었으므로,
새 리팩터링은 이 철학을 유지해야 한다.

---

## 사용자 의도 / 최종 UX 목표

사용자가 원하는 흐름은 아래다.

1. 기준 백테스트 실행
2. 전략 후보 셋 탐색
3. 전략 후보 셋을 리스트로 확인
4. 후보별로 비교
   - 수익률
   - PF
   - MDD
   - 거래 수
   - 신뢰도
   - 복합 점수
5. 특정 후보를 선택해서 재검증
6. 괜찮으면 저장
7. runtime 반영

즉 `전략 검증 랩`은 다음 화면이어야 한다.

> **탐색된 전략안 후보들을 보고, 그중 하나를 채택/보류/반영하는 콘솔**

종목별 승인 흐름이 끼어들면 안 된다.

---

## 반드시 지켜야 할 방향

### A. 종목 후보 개념을 다시 부활시키지 말 것
과거에는 아래가 섞여 있었다.
- `symbol_candidates`
- symbol approval
- symbol save
- symbol runtime apply

이 흐름은 이미 제거 방향이 결정되었고,
다시 전략 검증 랩에 끼워 넣으면 안 된다.

### B. runtime apply는 전략 후보 기준이어야 함
즉 runtime apply는:
- 저장된 전략 후보가 있으면 가능
- 종목 candidate가 없다고 막히면 안 됨

### C. `전략 후보 셋`은 실제 후보 리스트여야 함
단순 요약/summary가 아니라,
실제로 선택 가능한 후보 리스트여야 한다.

## 필요한 작업

## 1) optimizer 저장 구조 확장
대상 파일 예시:
- `apps/api/scripts/run_monte_carlo_optimizer.py`
- 필요 시 관련 service/store

### 해야 할 일
optimizer 결과 저장 시 아래와 같은 `strategy_candidates` 구조를 함께 저장한다.

예시 형태:

```json
{
  "strategy_candidates": [
    {
      "candidate_key": "cand-001",
      "label": "후보 1",
      "selection_source": "high_only",
      "patch": {
        "stop_loss_pct": 3.0,
        "take_profit_pct": 7.0,
        "max_holding_days": 14
      },
      "patch_lines": [
        "stop_loss_pct: 3.0",
        "take_profit_pct: 7.0",
        "max_holding_days: 14"
      ],
      "metrics": {
        "composite_score": 12.4,
        "profit_factor": 1.38,
        "validation_sharpe": 1.11,
        "trade_count": 71,
        "max_drawdown_pct": -18.5,
        "reliability": "high"
      },
      "summary": "..."
    }
  ]
}
```

### 핵심
- 후보 하나 = 전략안 하나
- 종목이 아니라 전략 파라미터 조합 기준
- 프론트가 그대로 렌더 가능한 구조

---

## 2) quant_ops workflow가 strategy candidates를 노출하도록 수정
대상 파일 예시:
- `apps/api/services/quant_ops_service.py`

### 해야 할 일
`get_quant_ops_workflow()` 의 `search_result` 안에
`strategy_candidates` 를 포함시키거나,
별도 top-level 필드로 노출한다.

예:
- `search_result.strategy_candidates`
또는
- `workflow.strategy_candidates`

둘 중 하나로 일관되게 선택.

### 추가 요구
- latest candidate가 어떤 strategy candidate에서 왔는지 식별 가능해야 함
  - `search_candidate_key`
  - `search_candidate_label`
- revalidate 시 특정 candidate를 지정할 수 있어야 함
  - 현재 임시 `candidate_key` 경로가 이미 일부 들어가 있을 수 있으니,
    최종적으로는 **진짜 strategy_candidates 기준**으로 정리할 것

---

## 3) 프론트 `전략 검증 랩`에서 진짜 전략 후보 셋 UI 구현
대상 파일 예시:
- `apps/web/src/pages/BacktestValidationPage.tsx`
- 필요 시 `apps/web/src/types/domain.ts`
- 필요 시 `apps/web/src/api/domain.ts`

### 해야 할 일
`상세 결과`의 `전략 후보 셋` 탭에서 아래가 보이게 한다.

#### 표시 컬럼 예시
- 후보 이름/ID
- 주요 patch 요약
- 복합 점수
- PF
- 검증 샤프
- 거래 수
- 최대 낙폭
- 신뢰도
- 선택/출처(high_only/medium_fallback 등)

#### 액션
- `이 후보로 재검증`
- 필요하면 `이 후보 저장`
- 현재 운영 후보와 동일하면 배지 표시

### UX 원칙
- 행 하나가 독립적인 전략안으로 보여야 함
- 사용자가 `이 후보로 운영 후보를 바꿔서 다시 검증` 할 수 있어야 함
- 단순 요약 카드가 아니라 **조작 가능한 리스트**여야 함

---

## 4) 운영 후보 판정 패널과 후보 셋을 자연스럽게 연결
### 목표
`전략 후보 셋`에서 특정 후보를 선택하면,
그 후보가 `운영 후보 판정` 패널의 최신 재검증 대상으로 연결돼야 한다.

즉 흐름은:

```text
전략 후보 셋에서 선택
→ 이 후보로 재검증
→ 운영 후보 판정 갱신
→ 운영 후보 저장
→ runtime 반영
```

### 주의
- 자동 latest candidate만 믿는 구조로 돌아가지 않게 할 것
- 사용자가 명시적으로 고른 후보가 최신 컨텍스트가 되어야 함

---

## 5) 기존 임시 다리 코드 정리
현재 일부 코드에는 임시로 붙인
- 후보 리스트 흉내
- 기존 구조 억지 매핑
- 모호한 라벨
등이 있을 수 있다.

전략 후보 셋이 정식 데이터 모델로 올라오면,
이 임시 연결 코드는 정리해야 한다.

---

## 이미 반영된 전제
현재 코드베이스에는 아래 방향이 이미 일부 반영돼 있을 수 있다.

- 종목 후보 관련 route 제거
- runtime apply의 종목 후보 의존성 제거
- 전략 검증 랩에서 종목 후보 UI 제거
- 결과 초기화 버튼 추가

새 작업은 이 전제를 유지한 채,
`전략 후보 셋` 부분만 제대로 완성해야 한다.

---

## 완료 조건
아래가 만족되면 완료로 본다.

1. `전략 후보 셋` 탭에 진짜 전략 후보 리스트가 보인다
2. 각 행이 독립 전략 후보로 이해 가능하다
3. 후보 하나를 골라 `이 후보로 재검증` 가능하다
4. 재검증된 후보가 `운영 후보 판정`에 반영된다
5. 저장/반영 흐름이 전략 레벨 기준으로 유지된다
6. 종목 후보 개념이 다시 끼어들지 않는다

---

## 사용자 관점의 기대 문장
최종적으로 사용자는 이렇게 느껴야 한다.

> "탐색된 전략 후보들을 보고,
> 그중 하나를 골라 운영 후보로 다시 검증하고,
> 괜찮으면 저장해서 runtime에 반영한다."

이게 이번 작업의 핵심이다.
